## 2026-07-09
**OCR-Automatik repariert — Uploads wurden nie automatisch verarbeitet** (Branch `feature/neuausrichtung-2026-07`, Commit `2497ae5e0`, systematic-debugging). Root Cause (gegen kombu-Quellcode verifiziert, NICHT „kein GPU"): `app/workers/celery_app.py` `broker_transport_options` hatte `priority_steps=range(10)` + `queue_order_strategy='priority'` → kombu legt pro `priority>0` eine Redis-**Sub-Queue** an (`ocr_high:9` etc.). Der GPU-Worker läuft `--pool=solo` und holt **nur Basis-Queues** → jeder normale Upload (`process_document_task`, Route-Priorität 9) landete in `ocr_high:9` und blieb ewig `pending`; ebenso Embedding (prio 8) u.v.m. **581.512 Beat-Periodics** hatten sich über Monate in `*:N`-Sub-Queues gestaut (Redis 595 MB → 17 MB nach Purge).
- **Fix**: `priority_steps` + `queue_order_strategy` entfernt (ohne `priority_steps` liefert `kombu._q_for_pri()` dank Default `[0]` IMMER den Basis-Queue-Namen). `sep=':'` **bewusst behalten** — es serialisiert AUCH die kombu-Binding-Tabelle (`_kombu.binding.*`); ein Entfernen brach die Bindings → `ValueError: not enough values to unpack` im `deliver→lookup`-Pfad → Worker crashte bei jedem Publish. Verseuchte Bindings gelöscht + sauber neu aufgebaut (Prozesse legen sie beim Queue-Declare neu an).
- **Live bewiesen**: normaler priority-9-Upload → Basis-Queue `ocr_high` → solo-Worker → **Surya-GPU-OCR** → `completed` (echter Lieferschein-Text, Umlaute korrekt) → Embedding (1024-dim) + RAG-Chunking laufen nach → volltext-durchsuchbar (tsvector). 0 Dokumente `pending`; alle Celery-Container healthy, Basis-Queues drainen unter Live-Beat-Last.
