# Design-Paket Migration 274 — `documents`-INSERT-Mandantentrennung (3-Policy-Verbund)

**Datum:** 2026-07-11 (früh) · **Status:** DESIGN (von Ben beauftragt; Umsetzung als eigener Schritt) · **Basis:** angewandte Migration 273, Audit `2026-07_rls_restrunde_audit.md` Punkt 2.

## 1. Problem

INSERT auf `documents` ist erlaubt, wenn IRGENDEINE permissive Policy-WITH-CHECK passt:

| Policy | WITH CHECK (Ist) |
|---|---|
| `documents_insert` (INSERT) | **`true`** — die offene Scheunentür |
| `documents_company_isolation` (ALL) | company-match ∨ `app.is_admin` |
| `documents_tenant_isolation` (ALL) | bypass ∨ **`company_id IS NULL`** ∨ company-match |

Folge: eine kontextlose App-Rollen-Session kann beliebige (auch fremd-company- oder NULL-company-) Dokumente EINFÜGEN, obwohl sie seit 272/273 nichts mehr LESEN kann.

## 2. Empirie (2026-07-11, live gemessen)

1. **0 Dokumente mit `company_id IS NULL`** im Bestand (4 Docs, alle mit company_id; 1 ownerlos = Spiegel).
2. **Privat-Suite ist NICHT betroffen**: sie schreibt in ihre eigene Tabelle `PrivatDocument` (`app/services/privat/document_service.py:495`), nicht in `documents` — der NULL-Escape ist für Privat NICHT tragend. (Einziger `documents`-Bezug in privat/: ein READ in `tax_assistant_service.py:285`.)
3. Alle kanonischen Ersteller setzen company_id: Import-Pfad (`imports/document_creation.py:224`), Mirror (`odoo_mirror_service.py:453` = connection.company_id), API-Upload (require_company-Kontext).

→ Es gibt KEINEN legitimen Produzenten von NULL-company-`documents`. Der NULL-Zweig kann ersatzlos entfallen (kein owner-gebundener Sonderpfad nötig).

## 3. Ziel-Design (Migration 274)

1. **`DROP POLICY documents_insert`** — ersatzlos; die beiden ALL-Policies decken INSERT vollständig und korrekt. (Downgrade: `CREATE POLICY documents_insert ON documents FOR INSERT WITH CHECK (true)`.)
2. **`documents_tenant_isolation` WITH CHECK** → `is_rls_bypass_enabled() OR company_id = get_current_company_id()` (NULL-Zweig raus; USING wurde bereits in 272 gehärtet — ALTER POLICY … WITH CHECK (…) ohne USING-Angabe lässt USING unberührt).
3. `documents_company_isolation` bleibt unverändert (company ∨ admin).

**Resultierende INSERT-Matrix:** Worker-Bypass ✓ · Company-Kontext (nur eigene company_id) ✓ · Admin ✓ · kontextlos ✗ · NULL-company ✗ · fremde company ✗. UPDATE-new-row unterliegt denselben WITH CHECKs (gewollt).
